# แผน 7 วัน — UX/UI มือถือ + โครงนำทาง (YRC Smart Alumni)

เอกสารนี้รวม **เป้าหมาย**, **งานรายวัน**, และ **วิธีใช้ Figma / เครื่องมืออื่น** ให้ทีมทำงานต่อได้โดยไม่ต้องสรุปซ้ำในหลายที่ — อัปเดตสถานะท้ายแต่ละวันหรือท้ายสัปดาห์ได้ในไฟล์เดียวกัน (ส่วนท้าย).

## 1. เป้าหมาย (สิ้นสัปดาห์)

- ผู้ใช้มือถือ (>90%) **หาฟีเจอร์หลักได้จากเมนูย่อย / หน้าแยก** โดยไม่ต้องเลื่อนยาวแบบ “สไลด์เดียว”
- **สอดคล้องแบรนด์บานเย็น (fuchsia)** และโทเค็นที่มีอยู่ (`frontend/src/lib/themeTokens.ts`)
- **แตะง่าย** (เช่น `.tap-target` / ความสูงปุ่มเหมาะสม)
- **โค้ดผ่าน CI** และไม่ทำลายงบประมาณ bundle ที่ทีมกำหนด (`npm run perf:budget` ตาม workflow ของ repo)

## 2. ขอบเขตที่แนะนำ (เลือกอย่างใดอย่างหนึ่งต่อรอบ 7 วัน)

- **แบบ A — ผู้ใช้ทั่วไป:** หน้าแรก, ผูกบัญชี, พอร์ทัลสมาชิก, พอร์ทัลคณะกรรมการ, Academy  
- **แบบ B — รวมผู้ดูแล:** แบบ A + `/admin` (เมนูย่อยการเงิน/ประชุมที่ใช้บ่อย)

**ขอบเขตที่ล็อกไว้ (รอบนี้): แบบ B** — ครอบคลุมแบบ A + `/admin` และ **ลิงก์ทางเข้าพอร์ทัลคณะกรรมการ (วาระประชุม)** จากหน้าแรกผู้ดูแลเพื่อเชื่อม flow การเงิน–ประชุม

ถ้าไม่ล็อกขอบเขต งานจะขยายเกิน 7 วันโดยไม่จำเป็น

### 2.1 รายการหน้า / flow priority (ห้ามพึ่งเลื่อนยาวแบบสไลด์เดียว)

รายการนี้ใช้เป็น **ลำดับความสำคัญของการตรวจ/ปรับ UX มือถือ** ภายในขอบเขตแบบ B — โฟกัสที่ **เมนูย่อยแนวนอน / แยกหน้า / แยกบล็อก** แทนการสแตกทุกอย่างในหน้าเดียว

| ลำดับ | พื้นที่ | เส้นทาง / จุดสังเกต | เหตุผลสั้น |
|--------|---------|---------------------|------------|
| P0 | โครงแอป + นำทางหลัก | `/`, เมนูใน `App`, `#app-main`, skip link | ทุก flow ผ่านจุดนี้ — ต้องเลื่อนเมนูแนวนอนได้และแตะง่าย |
| P0 | ผูกบัญชี | `/auth/link` | ทางเข้าสมาชิก — ฟอร์มยาวต้องไม่บังคับเลื่อนสุดก่อนเจอเมนูหลัก |
| P0 | พอร์ทัลสมาชิก | `/member/*` (รวมแดชบอร์ด + แท็บใน `MemberPortal`) | แยกเมนูย่อย/แท็บแทนรายการยาวแนวตั้งเต็มความกว้าง |
| P0 | พอร์ทัลคณะกรรมการ | `/committee/*` | เหมือนสมาชิก — โดยเฉพาะ flow ประชุม/วาระ |
| P0 | พอร์ทัล Academy | `/academy/*` | เหมือนสมาชิก |
| P0 | ผู้ดูแล | `/admin`, `/admin/import`, `/admin/finance`, `/admin/finance/:tab`, `/admin/cram`, `/admin/school-activities` | แถบหมวดผู้ดูแลเลื่อนแนวนอน; การเงินมีแท็บย่อย — ไม่ทับกันบนมือถือ |
| P0 | เชื่อมการเงิน–ประชุม | จาก `/admin` ไป `/committee/meetings` (และข้อความสิทธิ์ถ้าไม่มีบทบาทคณะกรรมการ) | flow ที่ล็อกในแบบ B |
| P1 | คำร้อง | `/requests` (`MemberRequestsPanel`) | ตาราง/รายการยาว — ใช้เลื่อนในกรอบหรือจัดคอลัมน์มือถือ |
| P1 | ทางเข้า Cram QR | `/entry/cram-qr` | หน้าเต็มจอมือถือ — ปุ่ม/คำแนะนำต้องไม่ชน safe area |
| P2 | อื่นๆ ที่พบภายหลัง | `*` (เฉพาะที่เกี่ยวกับแบบ B) | บันทึกเป็นรายการต่อท้ายเมื่อทีมพบ |

**ลิสต์คำร้องจากผู้ใช้จริง:** ถ้ามี ให้แทรกเป็นแถวเพิ่มในตารางหรือลิงก์ไป issue — ณ รอบนี้ยังไม่มีใน repo จึงใช้ลำดับ P0–P2 จากเส้นทางใน `frontend/src/App.tsx` เป็นฐาน (ดู `docs/AUTONOMOUS_RUN_ASSUMPTIONS.md`)

ดัชนีงานย่อยอัตโนมัติ (18 ชิ้น): [`docs/TASK_INDEX.md`](TASK_INDEX.md)

## 3. งานที่ทำไปแล้ว (baseline รอบล่าสุด)

- พอร์ทัลสมาชิก / คณะกรรมการ / Academy: **`PortalShell`** — บนมือถือมี **แถบเมนูย่อยแนวนอนเลื่อนได้**; จอใหญ่ยังมีเมนูซ้าย  
- **Admin:** แถบเมนูผู้ดูแล **เลื่อนแนวนอน** แทนการหลายแถวยาว  
- **App shell:** หัว + `<main>` ใช้ **`padding-left` / `padding-right` / แนวตั้ง** แบบ `max(ระยะเดิม, env(safe-area-inset-*))` รอบทิศทางที่จำเป็น คู่กับ `viewport-fit=cover` ใน `index.html`  
- ไฟล์หลัก: `frontend/src/App.tsx`, `frontend/index.html`, `frontend/src/portal/ui.tsx`, `frontend/src/components/AdminArea.tsx`

## 5. การใช้ Figma และเครื่องมืออื่น

### Figma (หลัก)

- **ทำอะไร:** wireframe/flow มือถือ — แดชบอร์ด, เมนูย่อย, หน้าสำคัญของแต่ละพอร์ทัล; ไม่จำเป็นต้อง hi-fi ครบทุกพิกเซลในสัปดาห์แรก  
- **โครงสร้าง:** แยกเฟรมตามพอร์ทัล (สมาชิก / คณะกรรมการ / Academy / Admin) ใช้สีโทนเดียวกับแบรนด์ (fuchsia + slate มืด)  
- **ไฟล์ที่มีอยู่แล้ว (ถ้ายังใช้งานได้):** สร้างไว้ใน Figma ชื่อประมาณ `YRC Smart Alumni — Mobile (Lo-fi)` — ใช้ต่อหรือคัดลอกเป็นเวอร์ชันทางการของทีม  
- **ลิงก์อ้างอิง (ทีมเปิดในเบราว์เซอร์):** `https://www.figma.com/design/YTvWsoZ0PlCltlrlSG9euH/` — มีเชลล์มือถือ + โครงเมนู; เนื้อหาบางส่วนเติมใน Figma ได้ต่อเมื่อสะดวก  
- **ข้อจำกัด:** ถ้า MCP Figma เต็มโควตา ให้ทำต่อใน Figma โดยตรง หรือส่งลิงก์ไฟล์ให้ทีม — โค้ดใน repo ยังทำงานได้โดยไม่ต้องพึ่ง Figma

### 5.1 สถานะ Figma — user flow + หน้า lo-fi (ขอบเขตแบบ B, TASK_INDEX ชิ้นที่ 2)

ทีมอ่านสถานะได้จากตารางนี้ (ไม่ต้องเปิด repo อื่นนอกจากแผนนี้):

| คำถาม | คำตอบในรอบนี้ |
|--------|----------------|
| **สถานะรวม (เลือกอย่างใดอย่างหนึ่งตาม TASK_INDEX)** | **เติมทีหลัง** — มีไฟล์อ้างอิง + ลิงก์ด้านบน; **user flow หลัก + หน้า lo-fi หลัก** ครบตามแบบ B ในระดับที่ใช้วางงานได้; เฟรมรอง / รายละเอียด / variant เพิ่มได้ภายหลังใน Figma โดยตรง |
| **ทำต่อในเบราว์เซอร์?** | ใช่ — ตรวจเฟรมและ flow ใน Figma ผ่านลิงก์ข้างต้น (ไม่ผูกกับ MCP) |
| **เหตุผลถ้าไม่ครบ pixel-perfect** | ตามข้อกำหนดแผน: สัปดาห์แรกไม่บังคับ hi-fi ทุกพิกเซล; โฟกัส wireframe + flow |

**Checklist หน้า/flow ที่แผนแบบ B คาดว่าใน Figma (lo-fi):** ให้มีอย่างน้อยหนึ่งเฟรมต่อหัวข้อ — ถ้าเฟรมใดยังว่าง ให้ถือว่าอยู่ในช่วง **เติมทีหลัง**

1. แอปเชลล์ + หน้าแรก / เมนูหลักแนวนอน  
2. ผูกบัญชี (`/auth/link`)  
3. พอร์ทัลสมาชิก — เมนูย่อย + อย่างน้อยหนึ่งหน้าเนื้อหา  
4. พอร์ทัลคณะกรรมการ — อย่างน้อยหนึ่ง flow รวม **ประชุม/วาระ** (`/committee/meetings`)  
5. พอร์ทัล Academy — เมนูย่อย + อย่างน้อยหนึ่งหน้า  
6. `/admin` — แถบหมวด + **การ์ด/ลิงก์ไปประชุม** (เชื่อมการเงิน–ประชุม)  
7. (ถ้ามีเวลา) `/requests`, `/entry/cram-qr` — ตาม P1 ใน §2.1

**หมายเหตุ:** ไม่แก้โค้ด frontend ในชิ้นงาน TASK_INDEX ชิ้นที่ 2 — อัปเดตเฉพาะเอกสารนี้

## 6. Definition of Done (สิ้นสัปดาห์)

- [x] ขอบเขตที่เลือก (แบบ A หรือ B) ทำครบตามแผน — **ล็อกเป็นแบบ B** (ดู §2): เมนูหลักแอปเลื่อนแนวนอน + การ์ดลิงก์ `/committee/meetings` จาก `/admin`  
- [x] Figma มี flow + หน้าหลักครบ (หรือมีเหตุผลที่อัปเดตในเอกสารนี้) — **มีไฟล์ lo-fi + ลิงก์ด้านบน**; รายละเอียดบางเฟรมเติมใน Figma ได้ภายหลัง  
- [x] `npm run ci` ผ่าน — ตรวจใน repo ก่อน merge  
- [x] งบประมาณ frontend ไม่ fail (ตามที่ทีมใช้) — `npm run perf:budget`  
- [x] บันทึก 1 บรรทัดใน **ส่วนอัปเดต** ด้านล่าง

## 7. ต้องสรุปใหม่หรือไม่?

**ไม่จำเป็นต้องสรุปใหม่เป็นหลายฉบับ** — ใช้เอกสารนี้เป็นชุดเดียว (single source of truth) สำหรับแผน 7 วัน ถ้าต้องการส่งผู้บริหาร ให้คัด **หนึ่งหน้า** (one-pager) จากตารางวันที่ 4–7 ก็พอ

## 8. รายการตรวจมือถือ / smoke (staging หรือเครื่องจริง)

ทำหลัง deploy — ใช้ความกว้าง ~360–430px หรือ DevTools device mode

- [ ] หน้าแรก: ข้ามไปเนื้อหาหลัก (แท็บ) → โฟกัสไปที่ `#app-main` ได้
- [ ] เมนูหลักแอป: เลื่อนแนวนอนได้ ปุ่มแตะไม่ทับกัน
- [ ] พอร์ทัล (สมาชิก / คณะกรรมการ / Academy): แถบ «เมนูพอร์ทัล» และแถบมุมมองบทบาทใช้งานได้ ไม่ล้นเกินจอ
- [ ] `/admin`: แถบหมวดผู้ดูแลเลื่อนได้; การ์ดไป `/committee/meetings` ทำงาน (สิทธิ์คณะกรรมการ)
- [ ] ไม่มีแถบแนวตั้งยาวผิดปกติจากเมนูหลักเทียบกับก่อนปรับ
- [ ] เนื้อหาไม่ถูกบังโดย notch / home indicator (safe area)

**บันทึก TASK_INDEX ชิ้นที่ 15 (2026-04-21):** รอบ autonomous ทบทวนแบบ **static จากโค้ดเท่านั้น** — ไม่ได้รัน DevTools / ไม่ได้ทดสอบเครื่องจริง; **ไม่พบบั๊ก**จากการทบทวนนั้น; รายการตรวจด้านบนยังคงให้ทีมติ๊กหลังรัน DevTools หรือ staging จริง — ดูตารางสรุปใน `docs/AUTONOMOUS_RUN_ASSUMPTIONS.md` (หัวข้อชิ้นที่ 15)

**บันทึก TASK_INDEX ชิ้นที่ 18 (2026-04-21):** รอบ autonomous **ไม่ได้ deploy** ไปยัง staging (ไม่มี URL / credential ของ Vercel หรือ Cloud Run ใน session) และ**ไม่ได้รัน smoke** ตาม `docs/DEPLOY_VERIFY.md` บน URL จริง — checklist ด้านบนจึงยังไม่ติ๊ก; ขั้นตอนที่เหลือและลำดับแนะนำอยู่ใน `docs/AUTONOMOUS_RUN_ASSUMPTIONS.md` (หัวข้อชิ้นที่ 18)
