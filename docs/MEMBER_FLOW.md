# Flow สมาชิก: ทะเบียน → LINE → ตรวจสอบ → แก้ไข / สมัครใหม่

## สิ่งที่ระบบทำได้หลังอัปเดตนี้

1. **LINE Login** → ได้ `line_uid` ในเบราว์เซอร์  
2. **ตรวจสอบและผูก** — กรอก **รุ่น · ชื่อ · นามสกุล** ให้ตรงแถวที่ Admin นำเข้า  
   - พบในทะเบียน → ผูก `line_uid` กับแถวนั้น → เปิด **แท็บหน้าสมาชิก** อัตโนมัติ (เก็บ snapshot ใน session)  
   - ไม่พบ → แสดง **สมัครใหม่ (คำร้อง)** กรอกฟิลด์เดียวกันตามหัวตาราง + ส่งคำร้อง  
3. **หน้าสมาชิก** — หัวข้อย่อย: เปลี่ยนแปลงข้อมูลสมาชิก (ใช้ `update-self`), กิจกรรมโรงเรียน, สมาชิกศิษย์เก่า, กิจกรรมสมาคม, โรงเรียนกวดวิชา — สี่หัวข้อหลังเป็นโครง placeholder รอพัฒนาต่อ (ทุน/บริจาค/แดชบอร์ด ฯลฯ)  
4. **บันทึกข้อมูลเพิ่มเติม** → `POST /api/members/update-self` (ต้องผูก `line_uid` แล้ว)

## API

| Method | Path | หมายเหตุ |
|--------|------|-----------|
| POST | `/api/members/verify-link` | ตอบ `member` ทั้งแถวเมื่อผูกสำเร็จ |
| POST | `/api/members/update-self` | Body: `{ "line_uid", "updates": { "เบอร์โทรศัพท์": "..." } }` |
| POST | `/api/members/register-request` | Body รวม `batch`, `first_name`, `last_name`, `line_uid` และหัวภาษาไทยอื่น |

## เมื่อจะอัปโหลดรายชื่อ “ชุดที่ถูกต้อง” แทนชุดทดสอบ

1. **สำรองข้อมูล** (ถ้าจำเป็น) จาก Supabase Table Editor หรือ Export  
2. แนวทางสะอาด: แท็บ **Admin** → **ลบสมาชิกทั้งหมด** (หรือ `DELETE /api/admin/members/all` ด้วย `x-admin-key`) แล้ว **นำเข้าไฟล์ใหม่**  
3. สมาชิกที่เคยผูก LINE ไว้แล้ว — แถวใหม่จะได้ `id` / ข้อมูลใหม่ — อาจต้อง **ผูก LINE ใหม่** หลังนำเข้า (หรือวางแผน migrate `line_uid` ด้วยสคริปต์แยกถ้าข้อมูลจำนวนมาก)

## ความปลอดภัยเบื้องต้น

- `update-self` อนุญาตเฉพาะฟิลด์รอง — **ไม่แก้ รุ่น / ชื่อ / นามสกุล** จากฟอร์มนี้  
- การแก้ข้อมูลต้องใช้ `line_uid` ที่ตรงกับแถวในฐานข้อมูลแล้ว
