## 1 — ล็อกขอบเขต + รายการหน้า priority

- **เป้าหมาย:** สรุปขอบเขตแบบ B และรายการหน้า/flow ที่ห้าม “เลื่อนยาวแบบสไลด์เดียว” ให้ชัด
- **ขอบเขตหลัก:** เอกสารใน `docs/` (เช่น บรรทัดสั้นใน `UI_7_DAY_PLAN_2026-04.md` หรือไฟล์ย่อยที่อ้างจากแผน)
- **เกณฑ์เสร็จ:** มีรายการ priority อ่านได้ในทีม; ไม่บังคับแก้โค้ด
- **ห้าม:** ขยายขอบเขตเกินแบบ B โดยไม่บันทึกเป็นงานใหม่

## 2 — Figma: user flow + หน้า lo-fi

- **เป้าหมาย:** ให้ flow + หน้าหลักครบตามแผน หรือบันทึกเหตุผล/ลิงก์ในแผน
- **ขอบเขต:** Figma ภายนอก repo + อัปเดตข้อความใน `docs/UI_7_DAY_PLAN_2026-04.md` ถ้าจำเป็น
- **เกณฑ์เสร็จ:** แผนอ้างอิงสถานะ Figma ได้ชัด (ครบ / เติมทีหลัง / ทำต่อในเบราว์เซอร์)
- **ห้าม:** บังคับแก้โค้ด frontend ในชิ้นนี้

## 3 — พอร์ทัลสมาชิก: โครง + เมนูย่อยมือถือ

- **เป้าหมาย:** `PortalShell` + เมนูย่อยแนวนอนบนมือถือใช้งานได้ ไม่ล้นเกินจอ
- **ขอบเขตหลัก:** `frontend/src/portal/ui.tsx`, `frontend/src/portal/memberPages.tsx` (และที่ import `PortalShell`)
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน
- **ห้าม:** เปลี่ยนพฤติกรรม API/backend

## 4 — พอร์ทัลสมาชิก: เนื้อหาแท็บ / ฟอร์ม

- **เป้าหมาย:** หน้า/แท็บเนื้อหาไม่กลายเป็นเลื่อนยาวเกินเมื่อไม่จำเป็น; แยกบล็อกชัด
- **ขอบเขตหลัก:** `frontend/src/components/MemberPortal.tsx`, `MemberProfileSection.tsx` และหน้าที่ member portal ใช้
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน
- **ห้าม:** refactor ใหญ่นอกพอร์ทัลสมาชิก

## 5 — พอร์ทัลคณะกรรมการ: โครง + เมนูย่อยมือถือ

- **เป้าหมาย:** เหมือนชิ้น 3 แต่เฉพาะคณะกรรมการ
- **ขอบเขตหลัก:** `frontend/src/portal/committeePages.tsx`, `frontend/src/portal/ui.tsx`
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน
- **ห้าม:** แก้กฎ RBAC ใน backend ในชิ้นนี้ (ยกเว้นเป็นข้อความ UI)

## 6 — พอร์ทัลคณะกรรมการ: หน้าลึก / รายละเอียด

- **เป้าหมาย:** หน้าที่ยาวมีโครงนำทาง/บล็อก ไม่เลื่อนยาวผิดปกติ
- **ขอบเขตหลัก:** คอมโพเนนต์ภายใต้ `committeePages` / เส้นทาง `/committee/*` ที่เกี่ยว
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน
- **ห้าม:** ขยายขอบเขตไป finance backend

## 7 — พอร์ทัล Academy: โครง + เมนูย่อยมือถือ

- **เป้าหมาย:** เหมือนชิ้น 3 แต่เฉพาะ Academy
- **ขอบเขตหลัก:** `frontend/src/portal/academyPages.tsx`, `frontend/src/portal/ui.tsx`
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน

## 8 — พอร์ทัล Academy: หน้าลึก

- **เป้าหมาย:** เนื้อหา/ฟอร์มยาวจัดระเบียบได้ ไม่เลื่อนยาวเกินเมื่อแยกแล้ว
- **ขอบเขตหลัก:** คอมโพเนนต์ภายใต้ `academyPages` / `/academy/*`
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน

## 9 — `/admin`: แถบหมวด + การ์ดไปประชุม

- **เป้าหมาย:** แถบหมวดผู้ดูแลเลื่อนแนวนอนได้; การ์ดไป `/committee/meetings` ชัดเจน (รวมข้อความสิทธิ์ถ้าจำเป็น)
- **ขอบเขตหลัก:** `frontend/src/components/AdminArea.tsx`, `AdminFinancePanel.tsx` (แถบย่อยที่เกี่ยว), ไฟล์ admin ที่เกี่ยวกับการ์ดลิงก์
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน

## 10 — ทวน flow Admin → `/committee/meetings`

- **เป้าหมาย:** สายทางและข้อความสอดคล้อง RBAC; บันทึกช่องว่างในเอกสารหรือ assumptions
- **ขอบเขต:** `RequireAppRoutes` / `RequireAppRoles` ที่เกี่ยว, ข้อความใน `AdminArea`, เส้นทางใน `App.tsx`
- **เกณฑ์เสร็จ:** มีบันทึกสั้นใน `docs/AUTONOMOUS_RUN_ASSUMPTIONS.md` หรือแผนหลักถ้าพบช่องว่าง; `npm run ci` ผ่าน

## 11 — App shell: เมนูหลัก + มุมมองบทบาท + safe area + skip-to-main

- **เป้าหมาย:** header/main เคารพ safe area; skip link ไป `#app-main`; เมนูหลักเลื่อนแนวนอน; แถบมุมมองบทบาทใช้บนมือถือได้
- **ขอบเขตหลัก:** `frontend/src/App.tsx`, `frontend/index.html`
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน

## 12 — `themeTokens` + `.tap-target` จุดสำคัญ

- **เป้าหมาย:** ปุ่ม/ลิงก์หลักใช้โทเค็นและเป้าแตะสม่ำเสมอ (ไม่ขยาย scope ทั้ง repo โดยไม่จำเป็น)
- **ขอบเขตหลัก:** `frontend/src/lib/themeTokens.ts`, `frontend/src/index.css`, จุดที่แตะบ่อยใน shell/พอร์ทัล
- **เกณฑ์เสร็จ:** `npm run ci` + `npm run perf:budget` ผ่าน

## 13 — หน้า entry สาธารณะ: หน้าแรก / ผูกบัญชี / QR cram

- **เป้าหมาย:** มือถือใช้งานได้; ไม่ล้น; แตะง่าย
- **ขอบเขตหลัก:** คอมโพเนนต์ที่ route `/`, `/auth/link`, `/entry/cram-qr` ใช้
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน

## 14 — คำร้อง `/requests` + `MemberRequestsPanel`

- **เป้าหมาย:** เลย์เอาต์มือถือและแถบเลื่อนไม่ล้น
- **ขอบเขตหลัก:** `frontend/src/components/MemberRequestsPanel.tsx`, เส้นทาง `/requests` ใน `App.tsx`
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน

## 15 — ทดสอบมือถือ DevTools (~360–430px) + บันทึกผล

- **เป้าหมาย:** รันทบทวนใน DevTools ตาม checklist ในแผน (ข้อ 8) เท่าที่ทำได้โดยไม่มี staging; บันทึกบั๊กหรือ “ไม่พบ”
- **ขอบเขต:** เอกสาร/บันทึกใน `docs/AUTONOMOUS_RUN_ASSUMPTIONS.md` หรือบรรทัดในตารางอัปเดตของ `UI_7_DAY_PLAN_2026-04.md`
- **เกณฑ์เสร็จ:** มีข้อความสรุปผลการตรวจ; ไม่บังคับแก้โค้ดถ้าไม่พบปัญหา
- **ห้าม:** อ้างว่าได้ทดสอบบนเครื่องจริงถ้าไม่ได้ทำ

## 16 — ภาษาไทย + ชื่อเมนู + empty/error

- **เป้าหมาย:** ข้อความว่าง/ผิดพลาดบอกขั้นตอนถัดไป; ชื่อเมนูไม่ขัดกันในแอปที่แตะบ่อย
- **ขอบเขต:** สตริง UI ในไฟล์ที่แตะในชิ้นก่อนหน้า (จำกัดไม่ refactor ทั้ง monorepo)
- **เกณฑ์เสร็จ:** `npm run ci` ผ่าน

## 17 — Gate: `npm run ci` + `npm run perf:budget`

- **เป้าหมาย:** ยืนยัน repo ผ่าน gate ตาม DoD ของแผน
- **ขอบเขต:** รันจาก root ของ monorepo
- **เกณฑ์เสร็จ:** ทั้งสองคำสั่ง exit 0; ถ้า fail ให้แก้เฉพาะที่จำเป็นจนกว่าจะผ่าน

## 18 — Deploy staging + smoke + อัปเดตแผน

- **เป้าหมาย:** ตามแผนวัน 7: deploy + smoke; อัปเดตส่วนอัปเดตใน `UI_7_DAY_PLAN_2026-04.md` และทำเครื่องหมาย checklist ข้อ 8 ตามที่ทำได้จริง
- **ขอบเขต:** เอกสาร + สคริปต์ที่ repo มี (`docs/DEPLOY_VERIFY.md` ฯลฯ) — **ถ้าไม่มี URL/credential** ให้บันทึกขั้นตอนที่เหลือใน `docs/AUTONOMOUS_RUN_ASSUMPTIONS.md` แทนการแกล้ง deploy
- **เกณฑ์เสร็จ:** เอกสารสะท้อนความจริง (deploy แล้ว หรือรอสิ่งที่ต้องมีจากทีม)
